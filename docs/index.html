E-Commerce
Microservices Backend

A production-style distributed e-commerce backend in Go — 6 services communicating over gRPC, async payments via Kafka, atomic stock management, and full Docker Compose containerisation. Same architecture patterns used by real e-commerce platforms.

SERVICES
Six Services, One System
api-gateway
Single entry point — translates REST to gRPC, validates JWT tokens at the edge. Internal services remain token-agnostic.
REST → gRPC
user-service
User registration and login. Issues JWT tokens on successful auth. Backed by PostgreSQL with GORM auto-migration.
JWT Issuer
product-service
Product catalog with atomic stock management. Uses SELECT FOR UPDATE inside a transaction — two concurrent orders for the last item cannot both succeed.
Atomic Stock
order-service
Orchestrates the order flow: validates product via gRPC, persists order, publishes order.created to Kafka. Client receives 201 immediately.
Kafka Producer
payment-service
Consumes order.created from Kafka asynchronously. Processes payment via mock gateway, checks for existing payment to prevent double-charges on retry.
Idempotent
notification-service
Consumes payment.processed and payment.failed topics. Sends email/SMS notifications for each outcome. Fully decoupled from payment logic.
Kafka Consumer

ASYNC ORDER FLOW
How an Order Becomes a Payment
POST
Client places order
HTTP POST /api/v1/orders hits the gateway. JWT is validated, request is translated to gRPC and forwarded to order-service.
POST /api/v1/orders → 201 Created {order_id}
01
order-service validates stock
gRPC call to product-service. Inside a database transaction with SELECT FOR UPDATE — atomic stock deduction prevents race conditions even under concurrent load.
SELECT FOR UPDATE → deduct stock → commit
02
Order saved, event published
Order persisted to PostgreSQL with status: pending. order.created event published to Kafka with full order payload. Client response returned immediately — no waiting for payment.
Kafka → order.created {order_id, user_id, product_id, amount}
03
payment-service consumes asynchronously
Kafka consumer picks up order.created. Before processing, checks for existing payment record — idempotent by design, safe to retry on Kafka redelivery without double-charging.
consume order.created → check idempotency → process payment
04
Payment result published
Payment outcome written to PostgreSQL. payment.processed or payment.failed event published to Kafka depending on result.
Kafka → payment.processed | payment.failed
05
notification-service sends alerts
Fully decoupled consumer on both payment topics. Dispatches email or SMS based on outcome. If notification-service is down, events queue in Kafka and are processed on recovery — no data loss.
consume → send email/SMS → ack offset
DESIGN DECISIONS
Why It's Built This Way
gRPC for internal communication
Services communicate using gRPC with a custom JSON codec. This gives strongly typed contracts via Protobuf — breaking changes are caught at compile time rather than at runtime. Far safer than ad-hoc REST between services.
compile-time safety
Kafka for async payment processing
Decoupling order creation from payment processing keeps the system responsive under load. If the payment service goes down, orders queue in Kafka and are processed on recovery — zero data loss. This is the same pattern used by Amazon and Flipkart at scale.
fault tolerance
Atomic stock deduction
The product service uses SELECT FOR UPDATE inside a database transaction to prevent race conditions. Two concurrent orders for the last item in stock cannot both succeed — the second transaction blocks until the first commits, then sees zero stock and fails cleanly.
consistency guarantee
Idempotent payment processing
The payment service checks for an existing payment record before processing. This makes it safe to retry failed Kafka deliveries without charging the customer twice. The check uses the order_id as the idempotency key.
exactly-once semantics
JWT validation at gateway only
Internal gRPC calls are trusted within the Docker network. Only the API gateway validates tokens, keeping internal services simple and fast. In production this would be hardened with mTLS for service-to-service authentication.
perimeter auth
